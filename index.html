浅谈thinkphp5.1的Facade原理与实现

Posted on 2018-03-28

最近几天在看ThinkPHP5.1（后面ThinkPHP简称tp）的一些东西，由于之前的几个项目主要是再用tp3在做，对于tp5没有了解过，昨天大概看了下tp5和tp3存在不少差异，感觉会更加友好，而且引入了很多新的特性，比如容器、Facade、依赖注入等等。
出于好奇，对于Facade的原理和使用进行一些简单的探索。
我们对于官方手册进行查阅后发现官方对于Facade的翻译为“门面”，其中对于Facade的说明还是很清晰的：

门面为容器中的类提供了一个静态调用接口，相比于传统的静态方法调用， 带来了更好的可测试性和扩展性，你可以为任何的非静态类库定义一个facade类。
系统已经为大部分核心类库定义了Facade，所以你可以通过Facade来访问这些系统类，当然也可以为你的应用类库添加静态代理。

这段话的意思其实很清晰，所谓的Facade是tp5.1为开发者提供的一套机制，开发者可以利用该机制为应用类添加静态代理，并且通过这些静态代理类实现静态调用类中的动态方法。为了理解，对于Facade设计模式，我们需要进行简单的介绍。

Facade设计模式

Facade设计模式，又被成为外观设计模式或者门脸设计模式。我们考虑一种情况，在某个场景下，我们需要对于很多类的对象进行一系列复杂的操作，比如说，当我们自己在家做午饭时，需要首先买菜，然后洗菜，然后烧饭，然后摆放碗筷，再吃饭，收拾碗筷，洗碗。如果能够将一系列的操作都简化成为一个统一的接口就叫作午餐流程。上面的流程过于繁琐，我们简单用A，B代表需要子系统的类，Facade类用于封装统一的接口提供给Client类进行调用。

代码如下：

class ClassA
{
    public function funcA1(){
        dump("This is ".__METHOD__);
    }
    public function funcA2(){
        dump("This is ".__METHOD__);
    }
}

class ClassB
{
    public function funcB1(){
        dump("This is ".__METHOD__);
    }
    public function funcB2(){
        dump("This is ".__METHOD__);
    }
}


class Facade
{
    public function DoIt(){
        $a = new ClassA();
        $b = new ClassB();
        $a->funcA1();
        $b->funcB1();
        $a->funcA2();
        $b->funcB2();
    }
}

class Client
{
    public function test(){
        $facade = new Facade;
        $facade->DoIt();
    }
}

可以看到，Client通过调用Facade类的DoIt方法实现了对于ClassA和ClassB的一系列操作。我们不需要关注其中的细节就可以完成一个完整的流程，只需要使用一个简单的函数，因此，Facade之所以被命名为Facade也就是因为它可以将一个类或者多个类的繁琐的细节以及关联都隐藏起来，只给调用者一个干净简单的外观。

Facade的方法本身并不进行功能的处理，Facade的方法的实现只是实现一个功能的组合调用。

我们最后来给出Facade设计模式的正式定义：

外观模式提供了一个统一的接口，用来访问子系统中的一群接口。外观定义了一个高层接口，让子系统更容易使用

TP5.1中的Facade使用

TP5.1的Facade设计模式和前面的简单示例有很大的不同，使用了静态代理等技术，但是它们的思想是一致的，就是给调用者尽可能简洁的的借口。
如果要对我们自定义的某个应用类使用Facade，那么我们需要做的事情很简单，假设我们的自定义类为Test类，里面有一个hello方法，那么

<?php
namespace app\common;

class Test
{
    public function hello($name)
    {
        return 'hello,' . $name;
    }
}

一般来说，如果调用这个类的hello方法，我们需要首先实例化一个对象，然后通过对象进行方法调用：

$test = new \app\common\Test;

echo $test->hello('thinkphp');

如果要用Facade静态代理，我们需要首先定义一个类：

<?php
namespace app\facade;

use think\Facade;

class Test extends Facade
{
    protected static function getFacadeClass()
    {
        return 'app\common\Test';
    }
}

这个类需要继承think\Facade，在定义之后我们就可以使用静态方式去调用第一个类的动态方法，如下：

echo \app\facade\Test::hello('thinkphp');

也就是说，TP5.1中的Facade主要是为了让使用者在不进行类实例化的情况下对于方法直接静态调用。

对于上面的getFacadeClass方法我们在其中定义需要进行静态代理的类。如果没有通过getFacadeClass方法显式指定要静态代理的类，还可以在调用的时候进行动态绑定：

<?php
namespace app\facade;

use think\Facade;

class Test extends Facade
{
}

可以看到代理类中没有任何方法，这是一个空类，只是继承了Facade类。为了能够获得代理类与被代理类之前的联系，我们需要在使用之前进行手动绑定。如下所示：

use app\facade\Test;
use think\Facade;

Facade::bind('app\facade\Test', 'app\common\Test');
echo Test::hello('thinkphp');

官方手册中还告诉我们系统给内置的常用类库定义了Facade类库，包括：APP、Build、Cache等。因此，我们在使用的时候并不需要进行自己手动绑定。

##TP5.1中的Facade实现方式

上面我们知道了如何使用Facade，但是TP5.1中对于Facade是如何实现的，下面进行简单的探究。
首先，对于从目录中找到了官方对于常用类库的Facade实现类：

打开任何一个facade目录中的静态代理类进行源码的查阅，都会发现这样一个事实：
所有的类的内容都是空，也就是说官方都是通过手动绑定的。比如：

namespace think\facade;

use think\Facade;

/**
 * @see \think\Config
 * @mixin \think\Config
 * @method bool has(string $name) static 检测配置是否存在
 * @method array pull(string $name) static 获取一级配置
 * @method mixed get(string $name) static 获取配置参数
 * @method mixed set(string $name, mixed $value = null) static 设置配置参数
 * @method array reset(string $prefix ='') static 重置配置参数
 */
class Config extends Facade
{
}

可以看到，这段源码中只有对于提供的接口的说明。具体的Config代理类仅仅继承了Facade，没有自己的方法。
那么，按照官方手册的说明，这种情况下需要我们对于代理类和被代理类进行手动绑定，进行源码查阅之后，我们可以发现在base.php文件中，具有一段绑定的代码，如下所示：

// 注册核心类的静态代理
Facade::bind([
    facade\App::class        => App::class,
    ...
    facade\View::class       => View::class,
]);

// 注册类库别名
Loader::addClassAlias([
    'App'      => facade\App::class,
    ...
    'View'     => facade\View::class,
]);

这和上文中我们自定义的注册方法其实是一致的，官方采用了批量注册的方法。

到这里，我们知道了常用的类库绑定是在base.php中实现的，而其他的部分和自定义的类库实现没有区别。

但是具体上来说，静态代理是如何是实现的呢？为什么我们通过设定一个绑定就可以采用静态方式调用动态方法了？我们发现每一个静态代理类都继承了Facade.php类，对于该类我们进行简单的源码分析：

<?php
namespace think;

class Facade
{
    protected static $bind = [];

    public static function bind($name, $class = null)
    {
        if (__CLASS__ != static::class) {
            return self::__callStatic('bind', func_get_args());
        }

        if (is_array($name)) {
            self::$bind = array_merge(self::$bind, $name);
        } else {
            self::$bind[$name] = $class;
        }
    }

    protected static function getFacadeClass()
    {}

    public static function __callStatic($method, $params)
    {
        return call_user_func_array([static::createFacade(), $method], $params);
    }

    ...
}

通过对于上面源码的解读，我们发现了在实现代理类的两种方法的机制：

  1. 如果实现getFacadeClass方法，那么将会覆盖Facade类的getFacadeClass方法，我们会直接使用该代理类的getFacadeClass方法，其他的方法进行调用时可以直接获得被代理类的类名。

  2. 如果实现getFacadeClass方法，那么调用之前需要通过调用Facade::bind()方法进行手动绑定，这个方法如上所示，主要是在进行将绑定的两个类放到$bind数组中，其他的方法进行调用时，需要从这个数组中检索获得被代理类的类名。

假设我们有一个类为User，我们采用User::getName()进行调用，但是这里的getName实际上是一个动态方法，理论上来说，我们应该采用如下的方式进行调用：

user = new User();
user->getName();

那么当我们采用静态调用时发生了什么呢？callStatic就发挥了作用，callStatic是一个魔术方法。

这个方法的主要的作用在于，当在静态上下文中调用一个不可访问方法时，__callStatic() 会被调用。

那么，Facade类重写了这个魔术方法，所以，当我们调用了一个静态代理类中事实上并不存在的静态方法时，将会执行这个函数，也就是说本质上执行用户自定义函数：call_user_func_array([实例, 方法],参数)，为了获得被代理类的实例，Facade又定义了一个获取对象的方法：
自定义静态方法createFacade()，这是一个专门获取代理类示例的方法，这个方法同样位于Facade类中：

protected static function createFacade($class = '', $args = [], $newInstance = false)
{
    $class       = $class ?: static::class;
    $facadeClass = static::getFacadeClass();

    if ($facadeClass) {
        $class = $facadeClass;
    } elseif (isset(self::$bind[$class])) {
        $class = self::$bind[$class];
    }

    if (static::$alwaysNewInstance) {
        $newInstance = true;
    }

    return Container::getInstance()->make($class, $args, $newInstance);
}

这个方法主要实现了静态代理类的手动创建。其本质是通过通过容器来实例化对象，在 createFacade 方法中，将会通过getFacadeClass得到具体被代理的类，getFacadeClass的机制我们上文已经分析过了。方法通过Container::getInstance()->make($class, $args, $newInstance)利用容器进行了对象的实例化操作，从而获得了一个被代理对象的实例。

简而言之，门面的实现是通过PHP的魔术方法 __callStatic，再配合容器来实现动态类的静态化调用。

TP5.1的Facade机制基本上就是这样一个原理，具体的实现还需要对于容器有一定的理解，其中还包括了反射机制以及依赖注入的使用，后面我会对于TP5.1中容器的内部实现机制进行学习。
